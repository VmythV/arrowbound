# arrowbound 状态机与边界规则

## 1. 文档目的

本文档定义场景切换、弹窗、挑战、奖励、刷新和异常恢复行为。目标是让同一个操作在任何状态下只有一个确定结果。

---

## 2. 顶层游戏状态

```ts
type GamePhase =
  | "boot"
  | "blessing_select"
  | "playing"
  | "challenge"
  | "reward"
  | "level_transition";

type ModalType = "shop" | "settings" | null;
```

规则：

- `GamePhase` 同一时刻只能有一个值。
- 商店和设置作为 `playing` 或 `challenge` 上方的模态层，不替代底层阶段。
- 同一时刻最多打开一个模态层，不允许弹窗嵌套。
- 祝福选择和奖励属于独立阶段，本身已经暂停主游戏，不再打开其他弹窗。
- 每次状态切换通过统一 StateController 执行，Scene 不直接修改其他 Scene 的状态。

---

## 3. 状态转换

```text
boot
  → blessing_select（当前关尚未选择祝福）
  → playing（当前关已有祝福）

blessing_select
  → playing（选择并保存祝福）

playing
  → challenge（主动开始挑战）
  → level_transition（前往其他已解锁关卡）
  → reward（存在待领取奖励）

challenge
  → reward（首次挑战成功且存在宝箱奖励）
  → playing（失败、取消或无奖励的重复挑战结束）

reward
  → playing（奖励入账并确认领取）

level_transition
  → blessing_select（目标关无祝福）
  → playing（目标关已有祝福）
```

---

## 4. 输入规则

- 鼠标左键点击主游戏有效区域时射箭。
- `Space` 作为键盘射箭键。
- 点击 UI 按钮、弹窗或其遮罩不会同时触发射箭。
- 射箭冷却中、主游戏暂停中或不处于 `playing/challenge` 时忽略射箭输入。
- 鼠标进入可拾取金币的命中区域时触发拾取，不要求点击。
- 一个金币进入拾取流程后立即加锁，避免鼠标和宠物重复拾取。

---

## 5. 暂停规则

以下情况暂停主游戏模拟：

- 打开商店
- 打开设置
- 进入祝福选择
- 进入奖励弹窗
- 浏览器页面进入隐藏状态

暂停内容包括：

- 弓摆动
- 箭矢移动与碰撞
- 金币掉落和拾取
- 机器人射击计时
- 宠物移动和拾取计时
- 玩家射箭冷却
- 挑战倒计时

UI 动画和当前弹窗继续更新。恢复时沿用暂停前的剩余时间，不重新创建计时器。

---

## 6. 关卡切换

- 挑战进行中禁止切换关卡，也不能执行普通通关操作。
- 存在未关闭弹窗时禁止切换关卡。
- 进入 `level_transition` 后立即停止射箭输入并回收全部箭矢。
- 场上未拾取金币通过统一金币入账接口自动收入钱包，但不计入挑战分数。
- 自动收入完成后立即保存，再卸载当前关卡。
- 切换完成后使用目标关配置重新创建靶子和运行时对象。
- 已普通通关关卡前往下一关不重复扣费。
- 未普通通关关卡必须检查当前钱包余额，再以一次原子操作完成扣费、标记通关和解锁下一关。

---

## 7. 挑战规则

### 7.1 开始

- 只有 `playing` 状态可以开始挑战。
- 开始时生成唯一 `challengeRunId`，挑战分数清零，时间设为 60 秒。
- 开始挑战前已存在的金币不绑定该 ID。
- 挑战中允许打开商店和设置，打开后暂停挑战。
- 挑战中购买的升级关闭商店后立即生效。

### 7.2 计分

- 挑战期间由玩家或机器人新生成的金币绑定当前 `challengeRunId`。
- 鼠标或宠物拾取金币时，钱包始终增加该金币价值。
- 仅当挑战仍在进行且金币 ID 匹配时，挑战分数增加该金币价值。
- 挑战分数达到目标后立即停止倒计时并判定成功。

### 7.3 结束与重复挑战

- 首次成功：生成挑战宝箱待领奖记录，立即保存，然后进入奖励状态。
- 失败：不扣金币、不修改完成状态，返回普通游玩。
- 已完成的挑战允许再次游玩，但不再生成挑战宝箱。
- 重复挑战仍显示成功或失败结果，并计入统计。
- 挑战期间刷新或关闭页面视为挑战取消，无惩罚；挑战运行时状态不写入长期存档。

## 8. 奖励原子性

存档增加：

```ts
type PendingReward = {
  id: string;
  source: "challenge" | "lucky_first_ten";
  levelId: number;
  type: "coins" | "shop_level" | "extra_blessing_choice";
  amount?: number;
  shopItemId?: string;
};

type RewardSaveData = {
  pendingExtraBlessingChoices: number;
  pendingRewards: PendingReward[];
};
```

发奖顺序固定为：

1. 抽取并构造 `PendingReward`。
2. 将待领奖记录追加到 `pendingRewards` 队列并写入存档。
3. 打开奖励弹窗。
4. 玩家确认后发放奖励。
5. 更新挑战宝箱领取状态或祝福触发状态。
6. 从队列移除已领取记录并再次保存。
7. 若队列仍有奖励，继续展示下一项。

游戏启动时若发现 `pendingRewards` 非空，优先按队列顺序恢复奖励弹窗，不允许重新抽奖。

## 9. 幸运首箭状态

每关需要额外保存：

```ts
type LevelSaveData = {
  // 其他字段省略
  luckyFirstTenRewardClaimed: boolean;
};
```

- 只有玩家手动射箭命中 10 环可以触发，机器人命中不触发。
- 每关最多触发一次，离开并返回关卡不会重置。
- 已领取后再次进入该关时，祝福的其他效果不变，但不会重复发奖。
- 奖励为金币小宝箱，价值为 `floor(本关普通通关目标 × 10%)`。
- 正常 10 环金币仍照常生成，不被小宝箱替代。
- 生成奖励前先写入 `pendingRewards`，确认领取后再将金币加入钱包并移除该记录。

---

## 10. 商店与免费升级边界

- 正常购买只能购买已经解锁且未满级的商品。
- 一次购买只提升一级，扣费和升级作为一次原子操作。
- 免费升级只从已解锁且未满级的商品中抽取。
- 免费升级不会触发购买扣费，但会触发后续商品解锁条件。
- 没有可升级商品时，免费升级奖励转换为当前关金币奖励。
- 所有商店等级在读取存档时限制到 `[0, maxLevel]`。

---

## 11. 世界对象生命周期

- 箭矢和金币只属于创建它们的关卡运行实例。
- 切换关卡后，不保留箭的位置、金币位置、宠物目标或机器人计时进度。
- 普通保存不保存场上箭矢和金币，只保存已经入账的钱包金币。
- 箭矢飞出画布扩展边界或完成命中表现后回收到对象池。
- 金币被拾取、关卡卸载或超过对象寿命后回收到对象池。
- 对象池回收时必须清除输入监听、目标锁和 `challengeRunId`。

---

## 12. 存档与异常恢复

- 存档 key 固定为 `arrowbound_save`。
- 关键操作立即保存：购买、普通通关、祝福选择、挑战成功、生成奖励、领取奖励。
- 普通金币变化使用最多 2 秒防抖保存；挑战成功前强制刷新钱包存档。
- 存档解析失败时保留原始损坏字符串到备用 key，再创建默认存档。
- 缺失字段按当前版本默认值补齐；未知字段忽略。
- 当前关卡不存在时回退到最高有效已解锁关卡。
- 商品等级越界时限制到合法范围。
- 金币为负数、非有限数或非整数时修正为 0。
- `normalCleared` 与下一关解锁状态冲突时，以已经解锁的更高关卡为准修复前置关卡状态。

---

## 13. 设置存档

存档增加：

```ts
type SettingsSaveData = {
  masterVolume: number; // 0～1
  musicVolume: number;  // 0～1
  sfxVolume: number;    // 0～1
  muted: boolean;
};
```

默认值：总音量 `0.8`、音乐 `0.6`、音效 `0.8`、不静音。设置变化立即保存。

---

## 14. 实现约束

- 随机数通过统一 RandomService 生成，测试中可以注入固定种子。
- 金币入账只能经过统一 `collectCoin` 或 `grantReward` 接口。
- 普通通关只能经过统一 `clearLevelAndUnlockNext` 接口。
- 奖励只能经过统一 RewardSystem，禁止弹窗直接修改存档。
- UI 只发送意图事件，不直接写 SaveData。
- 所有计时使用 Phaser 游戏时钟或统一时钟服务，不使用分散的浏览器 `setTimeout`。
